API Reference
모든 API는 개발자님이 원하시는 대로 요청할 수 있습니다만, 염두에 두어야할 것이 몇 가지 있습니다.

리턴코드 안내
코드	내용
200	모든 데이터가 정상적으로 반환됨(레코드가 없을 수 있음)
204	레코드가 없음.
400	요청한 파라미터 중 누락되었거나 잘못되었을 수 있음. 개선하여야할 파라미터는 error_parameters에 배열로 반환됨.
401	권한을 벗어난 요청
402	하루 요청횟수를 초과하였을 때. 업그레이드 안내 페이지로 이동시켜야함.
405	인증토큰이 만료되거나 잘못 되었을 때.
※ 언제든지 사용자가 후잉에서 인증토큰의 권한을 취소할 수 있으므로 항상 이 값을 확인하여 재인증을 요구하는 구조여야 함.
500	서버측에서 발생한 에러. 잠시 후 다시 시도하면 됨.
요청횟수 제한
일일 요청 API는 개발시에도 이용할 때와 마찬가지로 유료결제 여부에 따라 제한이 됩니다. 만약 여러분이 엄청나게 좋은 앱을 개발하는데 후잉의 도움이 필요하시다면 support@whooing.com으로 연락을 주십시오. 우리는 좋은 파트너가 될 수도 있으므로 환영하는 마음으로 API 제한을 풀어드리겠습니다.

여러분의 앱을 사용하는 모든 사용자들은 지정된 횟수의 API만을 사용하게 되므로 다음과 같은 가이드를 따라 주시기 바랍니다.

모든 API 응답의 rest_of_api 부분을 체크하여 사용자에게 남은 API 횟수를 항상 알려줍니다.
응답코드가 402로 온 경우에는 '하루 할당 API가 종료되었으니 프리미엄으로 업그레이드 하시거나 다음날 이용해 주시기 바랍니다'라는 메시지를 띄워주십시오. 프리미엄으로의 업그레이드는 단순히 후잉의 수익을 증대하는 것을 넘어서 당신의 앱이 보다 더 유익하게 사용되도록 하는 조치이기도 합니다.
만약 당신의 앱을 사용하는 모든 사용자에 대해 무제한의 API나 기간제한 없는 서비스를 제공해야 한다면 support@whooing.com으로 별도의 협력메일을 보내주십시오. 별도로 협력 계약을 맺어 수익을 분배할 수 있습니다.

요청헤더
후잉의 요청은 x-www-form-urlencoded 방식을 사용하고 있습니다. get이 아닌 요청을 전달할 때에는 요청헤더에 `Content-Type: application/x-www-form-urlencoded`를 명시하여 주십시오.

캐싱하여야할 API들
기본적으로 모든 보고서들은 필요할 때마다 요청하지만, 몇 가지 API들은 항상 필요하지만 변동이 거의 없는 경우도 있습니다. 이러한 API들은 매번 요청을 하여 사용자의 잔여 API를 깎아먹기 보다, 한 번 요청했을 때 앱 메모리나 DB에 결과값을 자체적으로 캐싱하고 이용하는 방법을 추천합니다. 약간의 변동이 생긴 경우 다시 API를 요청하여 최신값으로 유지하여도 되고 직접 객체(배열)값을 수정하여 최신의 값을 유지하여도 상관없습니다.

다음의 API들이 해당됩니다.

API 종류	설명
user	유저정보이며 유저속성이 바뀌지 않는한 유효
sections	섹션리스트로 섹션이 바뀌지 않는한 유효
accounts	항목리스트로, 섹션이 바뀌거나 항목변동이 생기지 않는 한 유효
요청 가능 포맷들
포맷	설명
.json_array	일반적인 JSON + 리스트형의 배열은 모두 array로 변환(추천)
.json	과거 호환용 JSON
정확성보다는 효율
후잉은 사용자들에게 정확성보다는 속도와 효율적인 환경을 만들고자 더욱 노력합니다. '재무데이터인데 오류가 있으면 어떻게하느냐'고 반문하실 수 있지만 우리는 법적인 증거로써 1센트까지 맞추려고 노력하지 않습니다. 거기에 노력을 기울일 바에 차라리 보다 더 속도감있고 재미있는 환경을 구축하고자 합니다.

API를 통해서 데이터를 수정/삭제할 때에 결과 code가 200으로 반환이 되었어도 수정이나 삭제가 되지 않을 수도 있습니다. 하지만 그러한 일은 0.1%의 확률로 일어날 것이고, 사용자들은 인터넷으로 데이터 주고 받으면 그 정도 오류 부득이하게 날 수 있다고 염두에 두고 있습니다. 따라서 일일이 수정이나 삭제의 결과값에 대응하여 무결성을 확보하기보다 그냥 API를 요청하고 code가 200으로 반환되었으면 완료된 것으로 처리하여도 상관없습니다.

## v0.9.0 클라이언트 변경 사항 (api/whooing.go)

### APIResponse 확장
- `Message string` — 응답 메시지
- `ErrorParameters json.RawMessage` — 400 에러 시 잘못된 파라미터 목록
- `RestOfAPI int` — 남은 API 호출 횟수. 모든 호출 후 `Client.LastRestOfAPI()`로 조회 가능

### 에러 처리
- `APIError` 타입 도입 (`api/errors.go`). `errors.As`로 추출 가능
- 429: 3초 대기 후 최대 3회 재시도
- 402: 재시도 금지, 즉시 에러 반환
- 405: 재시도 금지, 즉시 에러 반환 (토큰 만료 → 재인증 유도)
- 500: 3초 대기 후 1회 재시도

### 캐싱 (`api/cache.go`)
- `GetSections()`: 5분 TTL
- `GetAccountsMap(sectionID)`: 5분 TTL (섹션 변경 시 `InvalidateCache` 호출로 무효화)
- `GetUser()`: 캐시 없음 (CLI 단명 프로세스라 TUI에서만 의미 있음)

### 파라미터 빌더 (`api/params.go`)
```go
p := api.NewParams().Section(cfg.SectionID).DateRange(start, end).Str("rows_type", "month")
// p.Values() → url.Values
```

### 토큰버킷
- 분당 20회, 버스트 최대 5회 허용
- CLI 단명 프로세스에서는 실질적으로 무효 (TUI 장기 실행 세션에서 효과)
